# MentorMatch Analytics Dashboard 🧭

A multipage **Streamlit** dashboard analyzing the MentorMatch UAE mentee dataset (1,334 rows × 32 columns — includes synthetic Fujairah / Umm Al Quwain rows so all 7 emirates are analysable) around **two deliberately separated business questions**:

1. **What drives subscription conversion** (Free → Basic/Premium)?
2. **What drives positive career outcomes** for mentees?

Live-deployable on **Streamlit Community Cloud** straight from this repo.

---

## 🚀 Quick start

### Run locally
```bash
git clone https://github.com/<your-username>/mentormatch-dashboard.git
cd mentormatch-dashboard
pip install -r requirements.txt
streamlit run app.py
```

### Deploy on Streamlit Community Cloud
1. Push this repo to GitHub (keep the folder structure below intact).
2. Go to [share.streamlit.io](https://share.streamlit.io) → **New app**.
3. Select the repo, branch `main`, main file **`app.py`** → **Deploy**.

The `data/MentorMatch_UAE_Clean.csv` file ships with the repo, so no secrets or external storage are needed.

## 🗺️ Emirate coverage
The original survey covered five emirates. Rows for **Fujairah (32)** and **Umm Al Quwain (16)** were generated by bootstrap-sampling profiles from the smaller existing emirates (light jitter, new IDs) so every UAE emirate appears in filters and maps. They are flagged transparently in the Tab 1 cleaning log as synthetic, not survey responses.

## ⚠️ Critical methodology notes (read before interpreting anything)

### Target leakage — handled by design
`career_outcome` (Positive/Limited) is almost perfectly separated by `income_growth_pct` (Limited ≈ −1.3% vs Positive ≈ +20.9%) and `goals_completed` (≈1.4 vs ≈3.1): the label was evidently **derived from** these fields. Consequences enforced throughout:

- Both columns are **excluded from the outcome classifier's features** (Tab 4). Including them would let a model rediscover the labeling rule and post ~100% "accuracy" that means nothing.
- `income_growth_pct` is instead the **regression target** (Tab 5) — the granular, information-rich version of the same signal.
- `confidence_score` is **borderline** (r ≈ 0.79 with income growth) — plausibly a self-reported co-outcome. Tab 4 exposes a toggle running the model **with and without it**; the observed AUC jump (~0.89 → ~0.95) is presented as evidence of its co-outcome character, not as a free win.

### Structural missingness — encoded, never imputed
All 706 Free-tier rows have NaN for `churned` and `days_to_subscribe` because those users **never subscribed** — the values are undefined, not unobserved. Median/mode imputation would invent churn behavior. Instead: `churned` gets an explicit `Never Subscribed` category, `days_to_subscribe` stays NaN by design, and churn is analyzed on the **580 paid users only** (folded into Tab 6, keeping the acquire→convert→retain funnel in one place).

### Minor cleaning (fully logged in Tab 1)
- `gender`: casing standardized (Male/MALE → Male)
- `emirate`: leading whitespace stripped (" Dubai" → "Dubai")
- `monthly_income_aed`, `avg_session_rating`: 9 missing each (<1%) → median-imputed, with the justification stated explicitly

### Model-choice justifications (summary)
- **Four classifiers compared** (KNN, Decision Tree, Random Forest, Gradient Boosting), each with a stated expected role — KNN as the curse-of-dimensionality victim, DT as the interpretable-but-overfitting baseline, RF as its variance-reduced fix, GB as the accuracy contender needing CV discipline at n≈1,286. All judged on **5-fold stratified CV** with accuracy, precision, recall, F1 **and** ROC-AUC (majority-class accuracy is ~55%, so accuracy alone is uninformative).
- **Regularization is motivated by VIF**, not applied as a checklist: `sessions_attended` VIF ≈ 5.6 with `messages_sent`/`months_active` as correlated engagement-volume proxies. Ridge/Lasso/ElasticNet then answer "shrink together vs select vs blend", with a live λ slider and shrinkage-path plots.
- **Clustering** uses behavioral features only (one-hot demographics would distort Euclidean distance — a demographic segmentation needs k-modes/Gower, named as a limitation), StandardScaler (ranges 0–24 vs 0–99), K chosen from **elbow + silhouette jointly**, PCA-3D with explained-variance disclosure, and a Ward dendrogram + ARI as an independent method check.
- **Apriori over FP-Growth**: ~13 categorical attributes × 1,286 transactions makes candidate generation trivial and levelwise rules business-readable; FP-Growth's advantage appears only at much larger scale.

### UAE labor-market context (sources)
The diagnostic tab interprets the Visa/Sponsorship challenge (206 mentees, concentrated in South Asian/African nationality groups) against the UAE's employer-sponsored residence system and its post-2021 reforms. Suggested primary sources for the write-up (verify current versions):
- UAE Federal Decree-Law No. 33 of 2021 on Regulation of Labour Relations (MOHRE)
- Cabinet Resolution No. 65 of 2022 — Green Visa / Golden Visa expansion (ICP/GDRFA)
- World Bank & ILO publications on GCC labor mobility and sponsorship (kafala-style) systems

### Limitations
Observational, cross-sectional data: every "driver" is a **predictor**, not a proven cause. Mentor match "predicting" income growth may reflect selection (motivated mentees both secure better matches and progress faster). The honest next step for any recommended lever is an A/B test.
